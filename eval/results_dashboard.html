Shadow read-path, bug hunting, bug fixing, and feature ideation — evaluated on real-world and synthetic benchmarks

TL;DR

ShadowFrog explores codebases without any bug reports, failing tests, or hints, building a persistent shadow knowledge base through iterative dream sessions (autonomous exploration sessions where the agent studies the codebase without any task, building up its .shadow/ knowledge base). We evaluate it in two acts: a mechanism study of the shadow's read path (§1), then a four-section impact tour across three task families (§§3–5), reported below as one mechanism row plus five impact rows (§5 contributes two rows, one per lens):

We begin with the foundation. The three impact studies that follow (§3–§5) all assume the agent can consult the shadow at task time. Before measuring what the shadow enables, we verify the read path itself — with controlled synthetic needles, so the ground truth is unambiguous: we know exactly what was planted, exactly what should be retrievable, and exactly which condition is being tested. This section uses a self-contained metric (needle recall, defined in Setup below); §2 then defines the metrics consumed by the impact studies.

Setup

no-shadow

Control. No notes at all — the assistant uses raw source files and grep only.

flat-shadow

All notes concatenated into one big SHADOW.md — no spatial structure.

shadow-frog

One markdown note per source file, mirroring the directory tree at .shadow/<path>.md. The layout we are advocating for.

ShadowFrog assumes an AI coding assistant accumulates project memory over time — captured both from the user (preferences, ad-hoc instructions, project conventions) and from the assistant itself (non-obvious behavioral contracts it noticed, edge cases, hard-won lessons from past bugs — the kind of thing a careful engineer would jot down so the next teammate doesn't have to re-derive it). We model each such memory as a single one-sentence "needle" attached to a specific function or class. The complete shadow knowledge base for a codebase is a pile of these needles, indexed by where they apply.

The experiment asks: when the assistant is later given a task that depends on those needles, can it find them again? We pose questions to claude-opus-4.6 about the django codebase under each of the three systems above; a second AI judges what fraction of the planted needles the assistant surfaces (= recall). Questions come in two flavors that mirror real assistant usage: path-known models edit-time recall ("I'm about to modify foo.py::bar — what should I know first?"), and path-unknown models Q&A recall ("a teammate is asking X — what relevant project knowledge do I have?").

We sweep five shadow sizes (small 50 needles, medium 400, large 2.5k, xlarge 10k, xxlarge 35k — modeling early-stage projects through long-lived ones) and seven tool-call budgets (1, 2, 4, 8, 16, 32, unlimited — modeling that every grep and file-read costs the assistant latency and money). Total: 69,342 assistant runs. Re-running the entire experiment on fastapi (~360 functions) reproduces the same patterns.

Scope note. The three-arm comparison (no-shadow · flat-shadow · shadow-frog) appears only in this section. The impact studies (§3–§5) use a simpler two-arm comparison: baseline (no shadow) vs ShadowFrog. The flat-shadow arm exists here specifically to isolate the contribution of the per-file layout — same shadow content, different organization — so the two findings below can be cleanly attributed.

What we found

1 Even with unlimited tool calls, no-shadow can't catch up

Track the purple line in every panel. It plateaus well below the green and orange lines and never closes the gap — even at the right edge of each panel, where the assistant has unlimited tool calls. On the headline cell (django · large shadow · path-unknown · unlimited budget), no-shadow recalls 67% versus shadow-frog's 99% — a +31pp gap that is not a budget artifact. It is a structural ceiling: pre-written shadow notes unlock recall that the assistant cannot reconstruct from the raw source alone, no matter how much it greps.

Wait — why is the purple line higher on path-unknown than path-known? Counterintuitive but real, and only true for no-shadow. The planted facts are shadow knowledge — by design they're not in the raw source. With path-known, the agent is told "the answer is in src/foo/bar.py", reads that file, finds nothing matching, and honestly returns "I don't know" on ~83% of questions (vs ~28% on path-unknown, large × b=∞). With path-unknown the agent greps broadly, picks up distributed evidence in comments and related files, and synthesizes a plausible answer — partly real reconstruction from scattered hints, partly confident hallucination — ceilinged around ~70%. shadow-frog and flat-shadow don't show this effect because the planted knowledge actually lives in the shadow, so being told the file path either helps or is neutral.

2 Layout becomes decisive as the shadow grows

Now compare shadow-frog (green) vs flat-shadow (orange) — same content, different layout. On the top row (path-known), shadow-frog stays at the ceiling across every budget and scale; flat-shadow caps at ~95% even with unlimited tool calls — its grep into a 6 MB monolith occasionally misses or returns ambiguous matches. On the bottom row (path-unknown), the two converge once the agent has plenty of greps. At small / medium / large shadow sizes (left three columns) the gap is mostly noise; at xlarge and xxlarge (right two columns) under tight budget, the per-file layout pulls dramatically ahead.

Practical implication. If your shadow is small (under ~1 MB) and your agents have generous tool-call budgets, the choice between per-file and concatenated layouts barely matters. As soon as the shadow grows past a few MB and you operate under a real tool-call budget, the per-file layout becomes essentially required.

From mechanism to impact. Having confirmed the agent can read its shadow back at realistic budgets — and that the per-file layout is what unlocks recall at scale — we now ask what that read path enables. Skim §2 for the metric definitions used by the dream-driven impact studies, then §§3–5 report the results: hunting and fixing real-world bugs (§3), bug hunting at scale against a strong baseline (§4), and imagining what to build next (§5).

Not every study confirms a lift — §3B finds that retrieval is not the bottleneck for bug fixing — but each study's diagnosis depends on knowing the read path itself is sound. That is what this section establishes.

One caveat to bridge. This section uses controlled synthetic needles so ground truth is unambiguous; that makes the numbers above an upper bound on retrievability. The impact sections (§§3–5) use real dream-discovered shadows — longer entries, dedup uncertainty, and cross-file structure — and effectively place a lower bound on the same read path.

Methodology

🛠 Scope. §1 was a read-path mechanism test, scored with needle recall (defined there). §§3–5 are all dream-driven impact studies. This section sets up both halves of that phrase: first, what a dream is (the unit being measured); then, the scoring rubrics that consume it — L1 / L2 / Judge for bug hunting (§§3A & 4), Resolved % for bug fixing (§3B), Lens 1 / Lens 2 for feature ideation (§5). Familiar with all of it? Skip to §3 — otherwise read this once, then refer back as needed.

A · The dream — what we're scoring

A dream is one autonomous exploration session: the agent picks an experiment, runs it on a worktree branch, and archives a discoveries manifest in .shadow/_dreams/. New dreams can fork off prior dreams, so chains compound across sessions. Everything in §§3–5 is downstream of a dream campaign — the metric tables below tell you how we measure those campaigns, the visualization here tells you what they look like.

Concrete example: the full lineage from one repo (pydata/xarray, n=115 dreams). Shape (depth, fork-out, compounding rate) is a property of every dream campaign in this dashboard; color marks the kind of work each dream explored (the same change-locality taxonomy used in §5's per-arm breakdown). The per-dream Lens-1 alignment score from Lens 1 below is surfaced on the click-pinned side panel for each slice that aligned with one or more maintainer-shipped features. Hover a slice for the dream’s proposal and lineage; click to read the full report.

What to take away. The dream is the unit measured everywhere else in this dashboard. When §3's per-task table reports dreams: 110, this is what those 110 look like as a population. When §4 says “knowledge compounds”, it means the outer rings here exist. When §5 reports a Lens-1 recall lift, click any slice to see which maintainer-shipped features that dream aligned with — those are what powered the lift. The xarray slice was chosen because it had the strongest Lens-1 result of the eight ideation repos — one concrete anatomy is worth more than eight averaged ones.

B · Scoring rubrics — how we measure dream-driven impact

Three task families, multiple metric families — all reported on a comparable scale within each section:

  • Bug Hunting (§§3A & 4) reports a coverage profile: L1 ⊇ L2 (nested string-match metrics) plus three independently-computed judge tiers (Judge ≥ AREA, Judge ≥ ADJACENT, Judge = IDENTIFIED).
  • Bug Fixing (§3B) uses a single binary outcome: did the agent's patch pass the official SWE-Bench grader?
  • Feature Ideation (§5) is evaluated through two complementary lenses: Lens 1 measures alignment to future-shipped features (1–5 ordinal score → recall@T), and Lens 2 blind-judges idea quality on its own merit across four evaluative dimensions (Groundedness, Insight, User Impact, Spec Clarity), plus a descriptive Change Size and a categorical Change Locality tag. All four evaluative dims are reported separately — not collapsed into a composite "quality" score.
Bug Hunting Metrics · §§3A & 4
L1 — File Hit (automated)
Did any agent finding reference a file modified by the gold patch? Pure string match on file paths; binary yes/no per bug. Measures whether exploration directs the agent to the right area of the codebase.
L2 — Function Hit (automated)
Did any finding reference the function/method changed by the gold patch? Parses diff hunks to extract symbol names and matches against agent reports. Measures depth of investigation, not just breadth.
Judge — LLM Verdict (Opus 4.6)
An independent LLM judge reads agent findings alongside the gold patch and rates each bug on a 4-level rubric (below; the underlying SWE-Bench scoring also tracks a fifth PARTIAL tier between IDENTIFIED and ADJACENT). Captures semantic understanding that string-matching misses — e.g., describing a bug in prose without naming the exact symbol.

LLM Judge — Verdict Rubric

IDENTIFIED
Exact bug found
Clearly describes the same bug or its direct manifestation. A developer could fix it from the report alone.
ADJACENT
Real bug in the same function
Found genuine issues in the same function/block as the gold patch — strong, just not the specific one being scored.
AREA
Right subsystem, wrong function
Explored the right file/module and showed understanding of the subsystem, but didn't drill into the bug's specific function.
MISSED
Unrelated findings
Findings unrelated to the bug's file or subsystem. The agent may have found real bugs elsewhere — just not at this target.
Bug Fixing Metric · §3B
Resolved % — Patch Pass Rate (official SWE-Bench grader)
Each task gives the agent the bug's issue text plus failing tests; the agent must produce a patch. The official SWE-Bench grader applies the patch in a clean container and runs the hidden test suite — binary pass/fail per task. Resolved % = mean across 3 seeds (150 runs per arm; 300 across both arms). This is an end-to-end outcome metric: it doesn't care how the agent found the fix, only whether the tests pass. Complementary to the hunting funnel above — hunting asks "did the agent locate the bug?", fixing asks "did the agent solve it?".
Feature Ideation Metrics · §5 (two lenses)
Lens 1 — Alignment to shipped features (recall@T)
For each repo, the agent generates 50 feature ideas; an LLM judge (Opus 4.6, batched per silver) scores each idea against every silver-shipped feature on a 1–5 ordinal alignment scale (1 unrelated → 5 exact match). Recall@T = fraction of silvers whose per-silver score reaches threshold T — reported under two framings (single-draw vs 3-seed pooled). Full rubric and framing definitions live next to the data in §5.
Lens 2 — Idea quality on its own merit (4 evaluative dims + Change Size + Change Locality)
A blind 3-arm judge (BL / SF / Human, source labels stripped) rates each idea on four evaluative dimensions — Groundedness, Insight, User Impact, Spec Clarity — on a 1–5 scale, plus one descriptive Change Size dim (1–5, single-file → architectural; not a quality signal) and one categorical Change Locality tag (one of 7: API Surface, New Feature, Internal Machinery, Cross-Cutting, Ecosystem Bridge, DX & Tooling, Operational Quality). Each evaluative dim is reported separately — not collapsed into a composite "quality" score — because the four dims tell genuinely different stories (e.g. SF wins big on Insight while losing on Groundedness; both signals matter on their own). 3,310 ideas × 3 judge models (Opus 4.6 primary, plus Opus 4.7 + GPT-5.5 for robustness) = 9,930 judge verdicts across 8 repos. Full rubric with hi/lo examples in §5.

With the dream unit and the scoring rubrics in hand, the first impact study applies the bug-hunting funnel (L1/L2/Judge) and the Resolved % metric to 50 real-world tasks from SWE-Bench Verified (§3).

Findings (Bug Hunting)

L1 file coverage is nearly always achieved (98%)
When the agent explores broadly, it almost always touches the right file. The harder problem is drilling deeper — L2 holds at 84%, but the LLM judge only counts 22% as a confident, fixable identification.
Strategy > volume
Dream count alone has no meaningful correlation with success (r² < 0.01 vs the 5-level judge score). The highest-volume task (scikit-learn-10908, 407 dreams) only reached ADJACENT, while pytest-7205 (95 dreams) and sympy-17655 (151 dreams) both hit IDENTIFIED — efficient targeted exploration beats brute-force volume.
Compounding works — in clear cases
Compounding sessions (where a later dream forks off an earlier dream's branch) demonstrably upgraded 5 tasks beyond what the first-pass dreams reached — including xarray-4356 and xarray-6461 (both upgraded to IDENTIFIED via flat-format reports) and sphinx-8548 (MISSED → ADJACENT). The flip side: a substantial fraction of compounding sessions still revisit prior ground without producing new discoveries — a known inefficiency we're tackling in the dream scheduler.
Prose-heavy reports undercount on automated metrics
For some repos (e.g., sqlfluff in the SWE-Smith eval), the agent describes bugs semantically without producing exact code references — L2 misses them but the judge catches them. Validates the need for an LLM judge.

Shadow Consumption Funnel

The pre-filter delivers shadow hints to 23/50 tasks, but the funnel collapses sharply at the consumption stage — the point where delivered shadow content gets opened, judged relevant, and used to change the patch. Only 1 of 50 tasks (pytest-7324) was demonstrably driven by shadow content — the rest of the +4.7pp comes from the structured investigate-first workflow.

Findings (Bug Fixing)

Strong baseline + narrow contestable zone
Copilot CLI alone resolves 77.3% — most "easy" bugs are within autonomous reach. Of the 9 tasks where outcomes can possibly diverge, SF wins 6 (67%). The +4.7pp absolute looks small only because the addressable space is small.
The win is mostly the workflow, not the shadow content
5 of the 6 SF-win tasks had no measurable shadow influence. Wins came from a structured investigate-first + completeness-verification prompt that makes the agent more thorough — independently of any KB.
Consumption is the bottleneck — not retrieval
The pre-filter delivers hints to 23/50 tasks. The agent opens them in only 12. Only 2 are relevant. Only 1 actually changes the outcome. The 23→12 gap is mostly rational skipping — a strong baseline often solves the task without needing the hint. Shadow shines on architectural / multi-site bugs (when read)

Pre-filter false positives are low-cost
21 of 23 delivered hints were irrelevant or empty, but no task was hurt — agents skipped or dismissed wrong hints quickly. Shadow is positive-EV even when it doesn't help. The safety case holds.
7 deterministic frontier failures — both arms fail every seed
Mostly astropy and seaborn tasks. Common traits: multi-file coordinated fixes, deep domain knowledge, subtle backward-compat. Targeted shadow content didn't help here either — this is the agent-capability frontier, not a shadow problem.

Leakage controls and exclusions — defenses, what slipped through, and what's next.

Built-in defenses (in place during this experiment). Each repository was prepared so the agent could not trivially read the answer: all test files (tests/, test_*.py, conftest.py, plus the manifest's fail_to_pass targets) were stripped, and .git was deleted then re-initialized as a single "initial state" commit — so git diff / git reset / git log cannot recover the unmutated source. (See setup_all_repos.sh & baseline/init_agents.sh.)

What slipped through (very rare). A handful of agents bypassed the defenses by running pip download / pip install against PyPI and diffing the clean upstream source against the mutated codebase — trivially recovering ~100% of injected bugs without any real code analysis. The numbers shown above already exclude these agents/dreams via CHEATER_EXCLUSIONS in budget_judge.py. Affected: BL pygments (agent_012), BL sqlglot (agent_032 / agent_054), SF astroid (dream t41), SF pptx (session9), SF sunpy (dream s5t10).

Going forward. All evaluation containers now launch with --network=none (see start_docker_env.sh & baseline/init_agents.sh), so PyPI leakage is impossible by construction in future runs.

The leakage-controlled strong-baseline lens. The baseline arm is the same agent with a near-identical prompt — just no .shadow/ available (the shadow prompt additionally instructs the agent to consult .shadow/; idea-side shadow_anchor fields are stripped before judging). Either arm still has the entire codebase, docstrings, READMEs, plus its pre-training prior over “what features popular Python libraries tend to ship.” Silvers are issues filed by users after our shadow snapshot and are not provided to either arm during ideation; the judge sees only title + description + subsystem (the source repo and arm are stripped — Lens 2 additionally shows files_referenced, see B-1 framing below). A meaningful lift here means shadow surfaces real user-facing directions that even a state-of-the-art LLM with full code access doesn't predict from priors alone.

1 — Unrelated
No meaningful overlap. The idea addresses a different problem.
2 — Same area
Touches the same subsystem or feature surface but doesn't address the silver's specific need.
3 — Clearly related
Addresses a closely related need a user filing the silver would recognize as relevant.
4 — Strong match
Clearly the same feature direction, plausible alternative phrasing or scope.
5 — Exact match
Functionally the same feature. A maintainer reading both would call them duplicates.
Two reporting framings · per-silver score aggregated over 3 seeds
Framing A Mean of seed-maxes — operational metric
Expected quality from a single 50-idea ideation run. For each (silver, seed): take the max score over that seed's 50 ideas. Average over 3 seeds. Recall@T counts a silver when its average best-of-50 score reaches T. Answers: "if I make one ideation call, what's the expected best match?"
Worked example: seed-maxes = [2, 3, 4] → A = 3, contributes to recall@T≥3.
Framing B Max of seed-maxes — scaling/ceiling view
Inference-time-scaling potential with 3× sampling. Take the max score across all 3 seeds × 50 ideas = pooled 150-idea draw. Recall@T counts a silver when its pooled max reaches T. Answers: "if I'm willing to sample 3× the budget, what's reachable?"
Worked example: seed-maxes = [2, 3, 4] → B = 4, contributes to recall@T≥4.

Findings — Lens 1 (Alignment to shipped features)

Shadow lifts recall at the “clearly related” bar — in both framings
+10.4 pp (single draw) and +8.0 pp (3-seed pool) at T≥3. The signal is strongest on architecturally deep repos (sympy, xarray, astropy, sphinx).
Shadow's strength is broad recall; baseline's is exact-match precision
Shadow has 47.8/50 broadly-relevant ideas per draw vs baseline 45.9/50; but baseline lands 4.75 strong matches per draw vs shadow's 3.92, and 2.6× more exact-matches overall. Different idea distributions; complementary strengths.
Shadow is more stable across seeds — baseline gets more from sample diversity
Median per-silver seed-stdev: 0.0 for shadow, 0.58 for baseline. This is why baseline gains more from framing-B pooling: its seeds contribute more independent draws, which is also why baseline's recall@T scales faster as the budget grows from 50 to 150 ideas.
Shadow uniquely catches 166 silvers at T≥3 (single draw); baseline catches 71
Symmetric difference at the “clearly related” bar. Under 3-seed pooling, shadow-unique rises to 178 and baseline-unique to 105 — reflecting baseline's scaling advantage on popular features.
Pylint and pytest flip to baseline wins under inference-time scaling
Both are small, well-scoped tools whose silver pool is dominated by famous popular-feature asks. Under framing B, baseline saturates that space with 150 draws (z = −3.02 pylint, −2.33 pytest). On the other 6 repos, shadow either still leads or is tied. A clean ceiling/saturation effect.
Shadow agent verifiably used the shadow — not just had it
Across the full population of shadow runs, 99.7% of generated ideas carry a non-empty shadow_anchor, and 99.8% of dream-id anchors point at a real dream directory. (One diagnostic sphinx trace in SHADOW_USAGE_ANALYSIS.md shows the shadow arm reading .shadow/ in 187/191 tool calls.) The lift is real, not a side effect of changed prompting.
Hybrid one-shadow-draw + one-baseline-draw would likely combine both strengths
The two distributions are systematically different (recall vs precision; broad vs deep; niche vs famous). A 1+1 ensemble would inherit shadow's high-recall floor on architecturally deep repos and baseline's exact-match peaks on famous-feature-gap codebases — a natural follow-on experiment.
Lens 2 · Idea quality (judged on its own merit)
What if we ignore the shipped issues entirely and just judge the ideas on their own merit? Lens 1 measured how often the agents anticipated real shipped features. But "the community filed an issue for it" is only one quality signal — and a backward-looking one. Lens 2 asks a different question: independent of any future ground truth, are these ideas actually any good? We blind-judge all 3,310 ideas — 1,200 baseline, 1,200 shadow, plus a third Human arm of 910 silver-shipped GitHub feature requests, formatted identically — on a rubric designed to disentangle what kind of idea was produced from how original or impactful it is: four evaluative dimensions (Groundedness, Insight, User Impact, Spec Clarity), one descriptive Change Size, and one categorical Change Locality tag. Single Opus 4.6 judge call per idea, arm-blind (judge sees only {repo, title, description, subsystem, files_referenced} — provenance, arm flag, and shadow anchors stripped); replicated by Opus 4.7 + GPT-5.5 (see Multi-Judge Robustness panel). A style-control audit, in which the 910 human issues are re-rendered into the agent-generic register and re-judged, is in the Style Control panel.

References real files, modules, classes, conventions, or known design constraints. 3 = names a real subsystem and demonstrates familiarity; 5 = pinpoints specific files/symbols and shows insight into how that part of the codebase actually works.

5

Change Size — What is the blast radius of this change? (descriptive, NOT a quality signal)

Breadth of code touched. 5 is NOT better than 1. A 1-file fix and an architectural rewrite are both legitimate ideas — Change Size just describes which kind. 1 = a single file, 2 = small handful of related files, 3 = multiple modules, 4 = cross-cutting infrastructure, 5 = architectural rewrite or new top-level subsystem. Reported as a distribution, used for the stratified analysis below; never folded into a per-arm "quality" score.

Exactly one of seven tags: api_surface (extends or refines a public API), new_feature (a user-facing capability not previously present), internal_machinery (refactor / structural change with no user-visible effect), cross_cutting (touches many modules under one theme), ecosystem_bridge (interop with another library / format), dx_and_tooling (developer-experience improvements: docs, tests, linting), operational_quality (perf, robustness, error handling, observability). Reveals the kind of work each arm produces — the single biggest arm-difference signal.

Method. Re-render all 910 human issues with claude-opus-4.6 using a faithfulness-controlled prompt (no fabrication; subsystem from GitHub labels first; files_referenced only from paths in body), then re-judge with all 3 judges. Groundedness gaps retain ~70%. User Impact gaps essentially unchanged (+0.39 SF, +0.19 BL after normalisation, both deepen slightly). Insight reverses for SF (humans become more novel than both agents). See Style Control panel above.

🐸 Bottom line, across both acts

Mechanism (§1) shows the symbol-anchored read path is structurally superior — 99% recall vs a 67% no-shadow ceiling, a +31 pp gap that holds even at unlimited tool calls. Impact (§§3–5) shows where that pays off and where the frontier moves: blind bug hunting compounds with dream effort (§§3A, 4); idea quality rotates toward Insight and User Impact (§5); and bug fixing exposes a different bottleneck — consumption of the shadow content, not retrieval of it (§3B). Where the shadow is read, it lifts results; the open work is delivering it more aggressively into the agent's prompt.

Thesis. A persistent, symbol-anchored shadow memory makes coding agents measurably better at finding bugs, modestly better at fixing them (consumption-bound), and meaningfully better at imagining features — with the read path demonstrably sound at 69K runs.
